# OAuth2.0 Extension for Multi-AI Agent Collaboration (OAuth2.0-AI-Agent)

**Status:** Draft | **Organization:** IETF

## Description
An extension to the OAuth 2.0 protocol designed for task-oriented multi-AI agent collaboration. It enables a lead agent to coordinate sub-agents as a task group by adding specific fields in token and message flows, streamlining the authorization process.

## Latest Summary
The **OAuth 2.0 Extension for Multi-AI Agent Collaboration** is an Internet-Draft proposing a specialized authorization framework for scenarios where multiple AI agents collaborate on complex tasks. In these environments, a "leading agent" coordinates several "sub-agents." To avoid the inefficiency of each sub-agent independently requesting tokens from an Authorization Server (AS), this extension allows a leading agent to act as an "Applier," requesting access tokens on behalf of the entire task group.

The specification identifies two primary collaboration policies:
1.  **Static Policy:** All tasks and sub-agents are determined upfront. The leading agent requests a consolidated access token for the group before execution.
2.  **Dynamic Policy:** Tasks and sub-agents are identified incrementally. The leading agent applies for tokens as needed and issues bound task credentials to sub-agents.

Key technical additions include new fields in the token and message flows, such as `applier_id` (used during the request) and `app` (a claim within the access token identifying the applier). It also introduces a new error code, `unauthorized_applier`, for cases where an agent lacks the registered capability to distribute tasks. The framework is designed to be fully backward-compatible with existing OAuth 2.0 workflows while supporting privacy mechanisms like selective disclosure for token transmission.

## What's New
The latest update, **version 01 (released February 2026)**, transitions the draft into an "Experimental" status. Key updates include a clearer distinction between Static and Dynamic task group authorization procedures. The revision refines the roles of the 'Applier' (leading agent) and 'Clients' (sub-agents) and clarifies the extension of the Client Registration process. Specifically, it introduces the requirement for a "capability" parameter in the leading agent's profile (e.g., "resolve intent and distribute tasks") to validate authorization requests. It also provides detailed examples of the access token structure, featuring the `app` claim and multiple `sbj`-`aud`-`scope` pairs to support task groups.
