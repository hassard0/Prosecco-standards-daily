# Agent Attachment Protocol (AAP)

**Status:** Draft | **Organization:** IETF

## Description
A network-centric mechanism that enables autonomous agents to securely attach to an overlay network infrastructure. It defines procedures for identity validation, capability exchange, and secure session establishment between agents and network edge nodes.

## Latest Summary
The **Agent Attachment Protocol (AAP)** is a proposed network-centric framework designed to facilitate secure, autonomous agent interaction within an overlay network. Developed as an Individual Internet-Draft at the IETF, the protocol aims to decouple agent application logic from physical network infrastructure. By introducing **Agent Gateways (GWs)** as domain boundary nodes, the protocol allows agents to maintain stable, location-independent identifiers (**AgentIDs**) while moving across different administrative domains or cloud environments.

The protocol's scope is strictly limited to the **Agent-to-Gateway (A2G)** attachment and access plane. It defines standardized procedures for discovery, mutual authentication, capability exchange, and session management. Once an agent is attached to a local gateway, the gateway manages a **Message Forwarding Table (MFT)** to route encrypted traffic to other agents via a network of interconnected gateways. This architecture abstracts the underlying IP topology, effectively creating a policy-controlled messaging substrate similar to an SD-WAN overlay but tailored for AI-driven autonomous entities. Key features include support for multi-homing, mobility, and various discovery mechanisms such as Anycast, DHCP, and DNS. Additionally, AAP remains agnostic to application-layer messaging formats like A2A or MCP, focusing solely on transport-level forwarding.

## What's New
The protocol was introduced in **March 2026** with the release of `draft-dunbar-agent-attachment-00`. This initial version establishes the foundational architecture for **Agent Gateways**, which serve as messaging routers that operate on transport headers without inspecting application payloads. Recent highlights include the definition of three specific gateway discovery mechanisms: **Anycast-based discovery**, **DHCP-based bootstrapping**, and **DNS-based resolution** via a directory service. It also introduces the **Message Forwarding Table (MFT)** concept, which allows gateways to map AgentIDs to local interfaces or next-hop gateways, ensuring stable communication even as agents change physical network locations.
