# Verifiable AI Refusal Events using SCITT (SCITT-AI-Refusal)

**Status:** Draft | **Organization:** IETF

## Description
This document defines a claim set for recording and auditing AI content refusal events. It leverages the SCITT (Supply Chain Integrity, Transparency, and Trust) framework to provide verifiable audit trails of AI refusal decisions, independent of serialization formats.

## Latest Summary
The **Verifiable AI Refusal Events using SCITT** standard defines a specialized claim set for auditing AI content refusal decisions. In the current landscape, safety mechanisms that refuse harmful or policy-violating requests often leave no permanent, externally verifiable record. This specification bridges that gap by providing a structured audit trail that leverages the **Supply Chain Integrity, Transparency, and Trust (SCITT)** architecture. 

The standard introduces a formal mechanism to record four core event types: **ATTEMPT** (receipt of a request), **DENY** (refusal of a request), **GENERATE** (successful generation), and **ERROR** (system failure). A critical feature is the **Completeness Invariant**, which requires that every logged attempt must eventually link to exactly one outcome (Deny, Generate, or Error), preventing providers from selectively logging only successful interactions. 

Technically, the claims are designed to be carried within SCITT Signed Statements and verified via SCITT Receipts. While the specification is serialization-agnostic, it providing normative CDDL definitions for CBOR and illustrative examples for JSON. To protect user privacy and intellectual property, the standard mandates the use of cryptographic hashes (e.g., `prompt-hash`) rather than storing the original raw prompt content. This ensures auditability for regulators and third-party auditors without exposing potentially sensitive input data. The standard is currently an Internet-Draft and is intended for informational status.

## What's New
The latest update (**v02**, January 2026) formalizes the relationship between refusal events and the SCITT architecture, specifically mapping events to **Signed Statement payloads** and AI systems to **Issuers**. Key additions include a refined **Completeness Invariant** to prevent selective logging and new requirements for creating an `ATTEMPT` record specifically *before* safety evaluation begins. The draft also references the **January 2026 Grok incident** as a primary motivation for why verifiable refusal records are necessary for regulatory oversight and external auditing. Implementation resources and a Risk Category Taxonomy (Appendix A) have also been expanded to provide clearer deployment guidance.
