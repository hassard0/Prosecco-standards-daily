# HTTP Message Signatures for automated traffic Architecture (Web Bot Auth Architecture)

**Status:** Draft | **Organization:** IETF

## Description
An architecture for identifying and authenticating automated HTTP traffic using cryptographic message signatures. It allows automated clients to sign outbound requests so that servers can verify their identity with high confidence.

## Latest Summary
The **HTTP Message Signatures for Automated Traffic Architecture** is a proposed standard aimed at identifying and authenticating automated web traffic (bots). Developed by Thibault Meunier and Sandor Major, the architecture leverages the existing **HTTP Message Signatures** framework to allow automated clients to cryptographically sign their outbound requests. This mechanism provides HTTP servers with a reliable way to verify the identity of a bot, distinguishing legitimate automated agents (like search engine crawlers or monitoring tools) from malicious or anonymous traffic.

The initiative addresses the long-standing challenge of "Web Bot Authentication," where servers often rely on brittle methods like IP whitelisting or easily spoofed User-Agent strings. By moving to a signature-based model, the standard aims to increase confidence in bot identity while reducing the operational overhead for both bot operators and site administrators. Recent discussions within the IETF `web-bot-auth` mailing list highlight a focus on defining specific use cases, such as search indexers and AI training crawlers, and carefully evaluating the potential "harms" or privacy implications of widely deploying a standardized bot identification framework. Currently, the project is hosted in a dedicated GitHub repository and is undergoing active technical refinement through the IETF community.

## What's New
The standard is currently in an active phase of **Requirements and Impact Analysis**. Within the last month (February 2026), the IETF `web-bot-auth` mailing list has seen a surge in activity focused on two critical areas:

1.  **Use Case Definition:** Participants like David Schinazi and Ted Hardie are refining the scope of which "bots" should be covered, focusing on automated agents that require elevated access or recognition.
2.  **Harm Assessment:** A significant thread led by Mark Nottingham is investigating the "Harms" of the proposal, specifically looking at how standardized bot signatures could potentially be used for tracking, fingerprinting, or exclusionary practices by servers.

Technically, the **latest draft revision** (published late 2025) has stabilized the integration with the base HTTP Message Signatures spec, shifting the focus from "how to sign" to the "architecture of trust" between bot owners and web hosts.
