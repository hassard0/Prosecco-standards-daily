# AI Agent Protocol (AI Agent Protocol)

**Status:** Draft | **Organization:** W3C

## Description
A protocol designed to address interconnection and interoperability between AI agents across different platforms. it focuses on standardized identity verification using Decentralized Identifiers (DIDs) and a structured mechanism for agents to publish their capabilities and interfaces.

## Latest Summary
The AI Agent Protocol is a developing W3C Community Group draft designed to enable interoperability between autonomous AI agents across different platforms and organizations. The protocol focuses on establishing a standardized framework for mutual recognition, trust establishment, and identity transfer. A central pillar of the specification is the use of Decentralized Identifiers (DIDs) to provide a verifiable identity primitive that does not rely on a central registry.

Key technical components include:
- **Identity Model**: Leverages W3C DID Core to ensure cross-vendor interoperability.
- **did:wba Method**: A "Web-Based Anchored" DID method that reuses existing Web PKI, DNS, and HTTPS infrastructure. This allows agents to resolve identities via standard URI structures (e.g., `did:wba:example.com:alice` maps to an HTTPS endpoint).
- **Authentication Flow**: A decentralized authentication mechanism where clients use private keys to sign HTTP request headers (`Authorization: DIDWba ...`). The service verifies these signatures by resolving the client's DID document and checking the public key.
- **Efficiency**: The protocol is designed to minimize handshakes; initial requests contain signature data, while subsequent interactions use standard JWT access tokens.

The draft currently identifies several areas as "TODO," including the detailed architecture overview and specific design goal refinements, indicating that while the identity layer is maturing, the higher-level agent communication patterns are still in early stages of standardization.

## What's New
The protocol has introduced the **did:wba** (Web-Based Anchored) method as a core identity mechanism. This recent focus emphasizes leveraging existing Web infrastructure (DNS/HTTPS) to resolve agent identities without the need for bespoke blockchain-based DID networks. Recent documentation updates have also added detailed specifications for the `Authorization: DIDWba` HTTP header, including requirements for JSON Canonicalization Scheme (JCS) normalization, nonces for replay protection, and the signature generation process using SHA-256 and private keys. The project has also integrated a specific ToC flattening script to improve the readability of the specification as it transitions into a W3C Community Group draft.
