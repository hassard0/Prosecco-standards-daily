# AI Agent Discovery and Invocation Protocol (AI-ADI)

**Status:** Draft | **Organization:** IETF

## Description
A standardized protocol for the discovery and invocation of AI agents, defining a common metadata format for agent capabilities and a unified RESTful invocation interface. It aims to facilitate cross-platform interoperability and the formation of multi-agent systems via capability-based matching and intent-based selection.

## Latest Summary
The AI Agent Discovery and Invocation Protocol (AIDIP) is a proposed IETF standard designed to enable cross-platform interoperability among autonomous AI agents. As AI agents become more specialized and widespread, the lack of a unified interface has created "siloed" ecosystems; this protocol seeks to lower integration costs and facilitate the creation of multi-agent collaborative systems.

The protocol is built around four primary pillars:
1. **Agent Metadata Specification:** A structured JSON Schema that defines an agent's identity (`id`, `name`), high-level `capabilities` (e.g., "translation"), and detailed `operations` with specific I/O schemas.
2. **Discovery Mechanism:** A registry-based system where agents can be found via attribute-based filters (tags, languages) or semantic natural language queries.
3. **Unified Invocation Interface:** A standard RESTful API for executing agent tasks using consistent JSON payloads, regardless of the underlying vendor or platform.
4. **Security and Trust:** Guidelines for authentication (API keys, OAuth 2.0, mTLS) and secure transport to ensure safe agent-to-agent and client-to-agent interactions.

The protocol leverages existing web standards such as JSON Schema and OAuth 2.0 to ensure broad compatibility and ease of adoption for developers.

## What's New
The latest revision (**draft-cui-ai-agent-discovery-invocation-01**), published on February 12, 2026, introduces a significant optional extension: the **Agent Semantic Resolution (ASR)** layer. 

This new layer enables intent-based agent selection, allowing a coordinator (or "Host Agent") to describe a task's intent and receive a list of candidate agents before committing to a specific discovery query or invocation. This semantic matching phase is designed to augment existing discovery mechanisms without breaking backward compatibility or changing the core RESTful invocation semantics. Additionally, the metadata schema has been refined to better support core fields such as `supported_languages` and enhanced authentication objects comprising OAuth 2.0 scopes and mTLS.
